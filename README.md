# แจ้งเตือนข่าว SET เข้า Telegram อัตโนมัติ

ระบบนี้จะคอยตรวจหน้า "ข่าวหลักทรัพย์" ของ SET
(https://www.set.or.th/th/market/news-and-alert/news)
ทุก 10 นาที (ปรับได้) แล้วส่งข่าวใหม่เข้า Telegram โดยรันฟรีบน GitHub Actions
ไม่ต้องมีเซิร์ฟเวอร์หรือเปิดคอมทิ้งไว้

## ภาพรวมไฟล์
- `monitor_set_news.py` — สคริปต์หลัก เปิดหน้าเว็บ อ่านข่าว เทียบของเก่า/ใหม่ ส่ง Telegram
- `requirements.txt` — ไลบรารีที่ต้องใช้
- `state.json` — เก็บรายการข่าวที่เคยแจ้งไปแล้ว (สคริปต์จะอัปเดตไฟล์นี้เอง)
- `.github/workflows/monitor.yml` — ตัวตั้งเวลาให้รันอัตโนมัติบน GitHub

## ขั้นตอนตั้งค่า (ทำครั้งเดียว)

### 1. สร้าง GitHub repository
1. เข้า https://github.com/new สร้าง repo ใหม่ (ตั้งเป็น Private ก็ได้)
2. อัปโหลดไฟล์ทั้งหมดในโฟลเดอร์นี้ขึ้นไป (รักษาโครงสร้างโฟลเดอร์ `.github/workflows/` ไว้ด้วย)

### 2. ใส่ Telegram Bot Token และ Chat ID เป็น Secret
ในหน้า repo ไปที่ **Settings → Secrets and variables → Actions → New repository secret**
เพิ่ม 2 ตัว:
- ชื่อ `TELEGRAM_BOT_TOKEN` — ค่าคือ token ของบอทที่คุณมีอยู่แล้ว
- ชื่อ `TELEGRAM_CHAT_ID` — ค่าคือ chat id ที่ต้องการให้ส่งข้อความไปหา

(ห้ามใส่ token/chat id ลงในไฟล์โค้ดตรงๆ เพราะถ้า repo public จะรั่วได้)

### 3. เปิดใช้งาน Actions และทดสอบรันครั้งแรก
1. ไปที่แท็บ **Actions** ของ repo → กด "I understand my workflows, go ahead and enable them" (ถ้าเจอ)
2. เลือก workflow ชื่อ **Monitor SET News to Telegram** → กด **Run workflow** เพื่อทดสอบรันด้วยมือก่อน
3. เข้าไปดู log การรัน ถ้าขึ้นว่า "พบข่าวใหม่ ... รายการ" และคุณได้รับข้อความใน Telegram แปลว่าใช้งานได้แล้ว
4. รันครั้งแรกข่าวเก่าทั้งหมดจะถูกนับเป็น "ใหม่" และส่งเข้า Telegram รวดเดียว (เพราะ state.json ยังว่างอยู่)
   ถ้าไม่อยากให้ยิงข่าวเก่าทั้งหมดในรอบแรก ให้รันสคริปต์นี้ 1 ครั้งบนเครื่องตัวเองก่อน (ดูข้อ 4)
   เพื่อสร้าง state.json ที่มีข่าวปัจจุบันอยู่แล้ว แล้วค่อย commit ไฟล์นั้นขึ้น repo ก่อนเปิด schedule จริง

### 4. (แนะนำ) ทดสอบรันบนเครื่องตัวเองก่อน 1 ครั้ง
```bash
pip install -r requirements.txt
playwright install chromium

export TELEGRAM_BOT_TOKEN="ใส่ token ของคุณ"
export TELEGRAM_CHAT_ID="ใส่ chat id ของคุณ"
export DEBUG=1   # ใส่เพื่อดู log ว่าดักจับ endpoint ข่าวถูกต้องหรือไม่

python monitor_set_news.py
```
ถ้ารันแล้วขึ้น "ไม่พบรายการข่าว" ให้ดู log จาก DEBUG=1 ว่าดักจับ JSON endpoint อะไรมาบ้าง
แล้วแจ้งกลับมาได้ เดี๋ยวช่วยปรับ field ที่ใช้เดาโครงสร้างข่าวในสคริปต์ให้ตรงขึ้น

หลังทดสอบผ่านแล้ว ให้ `git add state.json && git commit && git push` ไฟล์ state.json
ที่มีข้อมูลข่าวปัจจุบันขึ้น repo ก่อน แล้วรอบถัดไปที่ Actions รันอัตโนมัติจะแจ้งเฉพาะข่าวที่ออกใหม่จริงๆ เท่านั้น

## แสดงชื่อย่อหุ้นในข้อความแจ้งเตือน
ข้อความ Telegram จะแสดงชื่อย่อหุ้นนำหน้าหัวข้อข่าวอัตโนมัติ เช่น `[PTT] แจ้งผลประกอบการไตรมาส...`
(ถ้าข่าวนั้นมีข้อมูลชื่อย่อหุ้นแนบมาด้วย — ข่าวบางประเภท เช่น SET News ทั่วไป อาจไม่มีชื่อหุ้นผูกอยู่)

อยากให้บางหุ้นได้รับ **ทุกข่าวที่ออกมา** (ไม่ต้องกรองหัวข้อเลย) ให้แก้ตัวแปร `SYMBOL_FILTER` ในไฟล์ `monitor_set_news.py`
(ตอนนี้ใส่ตัวอย่างหุ้นขนาดใหญ่ที่เป็นที่รู้จักทั่วไปไว้ให้แล้ว 50 ตัว เช่น PTT, AOT, CPALL, KBANK, ADVANC ฯลฯ — ไม่ใช่รายชื่อ SET50 ทางการ เป็นแค่จุดเริ่มต้น แก้ตามหุ้นที่คุณสนใจจริงๆ ได้เลย):
```python
SYMBOL_FILTER = ["PTT", "AOT", "CPALL", ...]
```
พฤติกรรมที่ได้:
- หุ้นที่ใส่ไว้ในลิสต์นี้ → ส่งเข้า Telegram **ทุกข่าว** ไม่ว่าหัวข้อจะเป็นอะไรก็ตาม
- หุ้นอื่นๆ ที่ไม่อยู่ในลิสต์ → ยังต้องผ่านตัวกรอง `TOPIC_KEYWORDS` ด้านล่างตามปกติ (เช่น เฉพาะข่าวงบการเงิน/ผลประกอบการ)
- ปล่อยเป็น `[]` = ไม่มีหุ้นพิเศษ ใช้ `TOPIC_KEYWORDS` กรองทุกหุ้นเท่ากันหมด

## การแจ้งเตือนตอนไม่มีข่าวใหม่ (รอบแรก / รอบสุดท้ายของวัน)
พฤติกรรมปัจจุบัน:
- **รอบทำงานทั่วไประหว่างวัน** (ไม่มีข่าวใหม่ที่ตรงเงื่อนไข) → **เงียบ ไม่ส่งอะไรเข้า Telegram** (ไม่รกทุก 15/30 นาที)
- **รอบแรกของวัน** ถ้ายังไม่มีข่าวใหม่เลย → ส่งข้อความ "👀 มารอดูกันว่า วันนี้จะมีข่าวอะไรใหม่" (ส่งแค่ครั้งเดียวตอนเช้า)
- **รอบสุดท้ายของวัน** (ชั่วโมงตาม `LAST_RUN_HOUR` ในไฟล์ ตอนนี้ตั้งเป็น 22 น.) ถ้าทั้งวันไม่มีข่าวเข้าเงื่อนไขเลยสักครั้ง → ส่งข้อความ "🌙 วันนี้ยังไม่มีข่าวอะไรใหม่" (ส่งแค่ครั้งเดียว)
- ถ้าดึงข่าวจากเว็บ SET ไม่ได้เลย (ปัญหาทางเทคนิค เช่นเว็บเปลี่ยนโครงสร้าง) → แจ้งเตือนเสมอทุกครั้งไม่ว่าจะรอบไหน เพื่อให้รู้ทันทีว่าบอทมีปัญหา

ระบบจดจำสถานะรายวันไว้ในไฟล์ `state.json` (รีเซ็ตอัตโนมัติเมื่อขึ้นวันใหม่ตามเวลาไทย)

**ข้อควรรู้:** "รอบแรกของวัน" และ "รอบสุดท้ายของวัน" อ้างอิงจากรอบที่บอทรันจริงๆ (ไม่ใช่เวลาที่ตั้งไว้ใน schedule)
ถ้าคุณกด "Run workflow" ทดสอบเองก่อนเวลารอบแรกตามปกติ ระบบจะนับรอบนั้นเป็น "รอบแรกของวัน" แทน — ถ้าไม่อยากให้ทดสอบมากระทบสถานะจริง
ให้ทดสอบตอนดึกๆ หลังผ่าน `LAST_RUN_HOUR` ไปแล้ว (ป้องกันชนกับรอบสุดท้ายของวันจริง) หรือรอวันถัดไป

ถ้าแก้เวลารอบสุดท้ายใน `.github/workflows/monitor.yml` (cron ของช่วงเย็น) อย่าลืมแก้ตัวแปร `LAST_RUN_HOUR` ในไฟล์ `monitor_set_news.py` ให้ตรงกันด้วย

**ข้อจำกัดเล็กน้อยที่ควรรู้:** ถ้าบังเอิญรอบสุดท้ายของวัน (22:xx น.) เกิดปัญหาดึงข่าวไม่ได้พอดี (fetch fail)
ข้อความสรุป "วันนี้ยังไม่มีข่าวอะไรใหม่" จะไม่ถูกส่งสำหรับวันนั้น (เพราะไม่มีรอบไหนหลังจากนั้นแล้ว) แต่ระบบจะยังส่ง
ข้อความแจ้งเตือน "ดึงข่าวไม่ได้" ให้ทราบว่ามีปัญหาแทน — กรณีนี้เกิดขึ้นได้ยากและไม่กระทบรอบแรกของวันถัดไป (แก้บั๊กแล้ว
ให้ fetch fail ไม่ไปแย่งสิทธิ์ข้อความ "รอบแรกของวัน" อีกต่อไป)

## กรองเฉพาะหัวข้อข่าวที่สนใจ (เช่น งบการเงิน ผลประกอบการ)
เปิดไฟล์ `monitor_set_news.py` แล้วแก้ตัวแปร `TOPIC_KEYWORDS` (อยู่ช่วงต้นไฟล์):
```python
TOPIC_KEYWORDS = [
    "งบการเงิน",
    "ผลประกอบการ",
    "งบไตรมาส",
    "กำไรสุทธิ",
    "Earnings",
]
```
- ใส่คำที่ต้องการกรองกี่คำก็ได้ ระบบจะแจ้งเตือนเฉพาะข่าวที่ "หัวข้อข่าว" มีคำใดคำหนึ่งในลิสต์นี้ (ไม่สนตัวพิมพ์เล็ก/ใหญ่)
- อยากดูทุกข่าวเหมือนเดิม ให้ใส่เป็นลิสต์ว่าง `TOPIC_KEYWORDS = []`
- ถ้าเว็บ SET ส่งฟิลด์ "ประเภทข่าว" มาด้วย (เช่น หมวด "ผลประกอบการ" ที่เห็นในตัวกรองบนหน้าเว็บ) ระบบจะเช็คฟิลด์นี้ประกอบด้วยอัตโนมัติ

## ใช้ cron ภายนอกแทน (แม่นยำกว่า) — แนะนำถ้า schedule ในตัวยังไม่นิ่ง
`schedule` ของ GitHub Actions เป็นระบบ "best effort" ไม่การันตีเวลารันจริง โดยเฉพาะช่วงที่ GitHub มีคนใช้เยอะทั่วโลก
อาจถูกเลื่อนคิวออกไปได้นาน วิธีที่แม่นยำกว่าคือให้บริการ cron ภายนอกฟรี (เช่น **cron-job.org**) เป็นคนยิงคำสั่งมาสั่ง GitHub ให้รันแทน

### 1. สร้าง GitHub Personal Access Token (PAT)
1. ไปที่ https://github.com/settings/tokens?type=beta → **Generate new token** (แบบ Fine-grained)
2. **Repository access** เลือก "Only select repositories" → เลือก repo นี้ repo เดียว (อย่าให้สิทธิ์ทุก repo)
3. **Permissions** → หา "Actions" → ตั้งเป็น **Read and write**
4. กด Generate token แล้ว copy เก็บไว้ (จะเห็นครั้งเดียว)

### 2. สมัคร cron-job.org (ฟรี ไม่ต้องผูกบัตร)
1. สมัครสมาชิกที่ https://cron-job.org
2. กด **Create cronjob** ตั้งค่าดังนี้:
   - **URL**: `https://api.github.com/repos/<ชื่อบัญชี>/<ชื่อ repo>/dispatches`
     (แทน `<ชื่อบัญชี>/<ชื่อ repo>` ด้วยของจริง เช่น `ckchet/set-news-today`)
   - **Request method**: `POST`
   - **Headers** (เพิ่มทีละอัน):
     - `Authorization` = `Bearer <PAT ที่ copy ไว้>`
     - `Accept` = `application/vnd.github+json`
     - `Content-Type` = `application/json`
   - **Body/Payload**:
     ```json
     {"event_type": "run-check"}
     ```
   - **Schedule**: ตั้งให้ตรงกับช่วงเวลาที่ต้องการ (cron-job.org คิดเป็นเวลาตาม timezone ที่เลือกได้ในบัญชี ตั้งเป็น Asia/Bangkok ได้เลย ไม่ต้องแปลงเป็น UTC เอง สะดวกกว่า native schedule ของ GitHub)
3. บันทึกแล้วกด **Run now** ทดสอบ 1 ครั้ง → เข้าไปดู tab Actions ของ repo ว่ามี run แบบ trigger "repository_dispatch" ขึ้นมาไหม

### ข้อควรระวัง (สำคัญ — อ่านก่อนเปิด native schedule กลับมาใช้)
- Token ที่ใส่ใน cron-job.org ให้สิทธิ์เฉพาะ repo นี้และแค่ Actions เท่านั้น ถ้า token รั่วความเสียหายจำกัดอยู่แค่ repo นี้
- **ห้ามเปิด native `schedule` ของ GitHub ใช้งานพร้อมกับ `repository_dispatch` จาก cron-job.org** ถ้าเวลาใกล้เคียงกันจะรันซ้อนกัน (concurrent runs)
  ทำให้ทั้งสองรอบอ่าน `state.json` เวอร์ชันเดิมพร้อมกัน (ยังไม่ทันเห็นว่าอีกรอบเพิ่งส่งข่าวไปแล้ว) แล้วส่งข่าวเดียวกันซ้ำเข้า Telegram ได้
  — ไฟล์นี้จึงปิด (comment out) `schedule` ไว้แล้ว เหลือใช้แค่ `repository_dispatch` เป็นหลัก
- ใส่ `concurrency` group ไว้ในไฟล์เป็นเกราะป้องกันอีกชั้น (กันไม่ให้รันซ้อนกันเด็ดขาดไม่ว่าจะมาจากไหน) และ commit step ปรับให้ `git pull --rebase` ก่อน push เผื่อกรณีมี commit อื่นแซงเข้ามา

## ถ้าเจอปัญหาข่าวซ้ำ (ข่าวเดียวกันถูกส่งเข้า Telegram มากกว่า 1 ครั้ง)
สาเหตุหลักคือ**มีการรันซ้อนกัน** เช็คตามนี้:
1. เข้า tab **Actions** ของ repo → ดูว่ามี 2 run เวลาใกล้เคียงกันมาก (ห่างกันไม่กี่วินาที-นาที) หรือไม่
2. ถ้าใช่ ให้เช็คว่า native `schedule` ถูก comment ปิดไว้จริงหรือเปล่า (ดูในไฟล์ `.github/workflows/monitor.yml`) และ cron-job.org ไม่ได้ตั้ง cronjob ซ้ำซ้อนกัน 2 อันโดยไม่ตั้งใจ
3. ถ้ายังเจออีกหลัง apply ไฟล์เวอร์ชันนี้แล้ว (มี `concurrency` group ป้องกันไว้) ให้เช็คว่า push step ล้มเหลวหรือไม่ (ดู log ขั้นตอน "Commit updated state.json") ถ้า push พังบ่อยๆ อาจเป็นเพราะ branch protection rule กันไว้ ต้องอนุญาตให้ `github-actions[bot]` push ได้

## ถ้าเจอ error "400 Bad Request" จาก Telegram
สาเหตุที่พบบ่อยที่สุดคือหัวข้อข่าวจริงบนเว็บมีอักขระพิเศษของ HTML (`&`, `<`, `>`) ปนอยู่
ทำให้ Telegram ตีความข้อความผิดรูปแบบ (เพราะสคริปต์ส่งแบบ `parse_mode: HTML`)
— ในเวอร์ชันนี้แก้ไขให้แปลงอักขระเหล่านี้ก่อนส่งอัตโนมัติแล้ว

ถ้ายังเจอ 400 อยู่ ให้เปิดดู log การรันใน tab Actions → หา บรรทัดที่ขึ้นว่า
`Telegram API error response: ...` จะมีข้อความจริงจาก Telegram บอกสาเหตุตรงๆ เช่น:
- `"chat not found"` → chat_id ผิด หรือบอทยังไม่ได้ถูกเชิญเข้ากลุ่ม
- `"bot was blocked by the user"` → ผู้ใช้บล็อกบอทไปแล้ว
- `"can't parse entities"` → ยังมีอักขระ HTML ที่ไม่รองรับหลุดไปอีก (แจ้งข้อความ error มาได้ ช่วยแก้ให้เพิ่ม)

## ปรับความถี่การตรวจสอบ
ตอนนี้ตั้งไว้ 3 ช่วงความถี่ต่อวัน (จันทร์-ศุกร์) ในไฟล์ `.github/workflows/monitor.yml` โดยจงใจเยื้องเวลาให้เริ่มที่นาทีที่ 7 แทนนาทีกลมๆ (ดูเหตุผลด้านล่าง):
- 06:07-07:37 น. ไทย → ทุก 30 นาที
- 08:07-16:52 น. ไทย → ทุก 15 นาที
- 17:07-22:07 น. ไทย → ทุก 30 นาที

แก้ตัวเลขได้ตามต้องการ (เวลาใน cron เป็น UTC เสมอ = เวลาไทย − 7 ชั่วโมง เช็คเทียบเวลาได้ที่ https://crontab.guru)

**สิ่งที่ควรรู้เกี่ยวกับ schedule ของ GitHub Actions:**
- GitHub ไม่การันตีว่าจะรันตรงเวลาเป๊ะๆ เป็นระบบแบบ "best effort" อาจดีเลย์หรือข้ามรอบได้บ้าง ถือเป็นข้อจำกัดของแพลตฟอร์ม ไม่ใช่ตั้งค่าผิด
- **นาทีกลมๆ (0, 15, 30, 45) มีคนตั้ง cron ชนกันทั่วโลกเยอะมาก** ทำให้คิวแน่นและถูกดีเลย์/ข้ามได้บ่อยเป็นพิเศษ — ไฟล์นี้เลยเยื้องเวลาให้เริ่มที่นาที 7, 22, 37, 52 แทน ช่วยให้รันตรงเวลาขึ้นได้มาก
- Repo แบบ **Private** จะถูกหักโควต้า "Actions minutes" ฟรี 2,000 นาที/เดือน (บัญชี GitHub Free) ต่อการรันแต่ละครั้ง ถ้าใกล้เต็มโควต้า ให้ลดความถี่ลงหรือเปลี่ยน repo เป็น public (public repo ใช้ Actions ฟรีไม่จำกัด)
- ถ้าเยื้องนาทีแล้วยังรันไม่สม่ำเสมอ ทางเลือกที่แม่นยำกว่าคือใช้บริการ cron ภายนอกฟรี (เช่น cron-job.org) ยิงไปเรียก GitHub REST API `workflow_dispatch` แทนการพึ่ง schedule ในตัวของ GitHub เอง (ต้องใช้ Personal Access Token เพิ่ม) บอกได้ถ้าอยากให้ช่วยตั้งค่าแบบนี้แทน

## ข้อจำกัดที่ควรรู้
- GitHub Actions แบบ schedule มีดีเลย์ได้บ้าง (ปกติไม่เกิน 2-3 นาที) ไม่ใช่ real-time เป๊ะๆ
- สคริปต์เดาโครงสร้างข้อมูลข่าวจาก JSON ที่เว็บ SET ส่งมาเองแบบอัตโนมัติ ถ้าทาง SET เปลี่ยนโครงสร้างเว็บ
  อาจต้องปรับ `TITLE_KEYS` / `DATE_KEYS` / `ID_KEYS` / `LINK_KEYS` ใน `monitor_set_news.py` ใหม่
- ถ้าต้องการกรองเฉพาะบางประเภทข่าว/บางหุ้น สามารถเพิ่มเงื่อนไขกรองใน `main()` ก่อนส่ง Telegram ได้ บอกได้เลยถ้าต้องการให้ช่วยเพิ่ม
